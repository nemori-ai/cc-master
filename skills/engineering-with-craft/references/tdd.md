## 铁律

```
没有失败的测试，不写产码。
```

这不是建议，是前置条件。

**若你已经写了产品代码才想起写测试：删掉，重来。** 不是「留着参考」，不是「一边看一边适配」——那是测试写在实现之后，换了个外套。删掉，意味着真的删掉；从失败测试出发，重新实现。沉没的时间无论如何都回不来，现在能选的只是：从失败测试长出的、你可以信任的代码，还是反向补写的、仅仅在交付时点碰巧通过的套路。

**若你没有亲眼看着测试失败，你不知道它在测什么。** 一个首次运行就通过的测试，只能证明它与它被写来讨好的实现是一致的——它可能什么边界都没覆到，可能压根连被测路径都没经过。

探索性代码是合法的例外——探索之后，把探索扔掉，从失败测试开始正式实现。把探索代码「改写成有测试的版本」是自我欺骗：你会调整测试去适配已经存在的实现决策，而不是让测试逼出更好的接口。

## 红绿重构循环

每一个行为变更，无例外地走完这五步：

### 1. RED — 写一个最小失败测试

一个测试，一个行为，一个清晰描述意图的名字。

- **先写期望的 API 调用和断言，再考虑如何实现**——测试是设计草图，不是实现的镜像。
- 用真实代码而不是 double，只在真正的进程边界（数据库、外部服务、消息队列）才引入替身。
- 不发明新的测试脚手架，先复用项目已有的 fixtures 和测试工具。

### 2. 验证 RED — 看输出，确认失败原因

**运行测试，读失败输出，确认它在你预期的点失败。** 这一步不能省。

三种失败状态，意义各不同：

| 测试状态 | 含义 |
|---|---|
| FAILED（预期原因）| 功能缺失，继续写最小通过代码 |
| FAILED（意外原因）| 测试本身有问题（导入错误 / 夹具缺失），修测试，不是修标准 |
| PASSED（首次运行）| 测试在测已有行为，不是新行为；或测试根本没有触达被测路径——删掉重写 |

**不要截断输出、不要用「看了一眼感觉应该失败」代替实际读日志。** 失败时的完整 traceback 里往往藏着你最需要的信息。

### 3. GREEN — 写最小通过代码

**只写让这个测试通过的最少代码，绝不多写。** 没有测试要求的参数、没有「将来可能用到」的分支、没有提前重构邻居——YAGNI 在这一步严格执行。

### 4. 验证 GREEN — 读输出，确认通过且无回归

- 新测试通过。
- 套件其余部分仍然通过。
- 没有新的 warning 或 stray error——工艺上，warning 是未来的 bug。

### 5. REFACTOR — 只在 green 上重构

消除重复，改善命名，提取 helpers；**不增加行为**。重构过程中保持 green：每一步小改动之后，运行测试确认没有破坏。

> **关于「不要反复运行测试」的规则**：RED 运行和 GREEN 运行是两个**语义上不同的验证**——它们就是 TDD 的要点，不是浪费。禁止的是：代码没变、套件没变，却反复运行来「安慰自己」。每次状态变化运行一次；读日志，不要用再次运行代替思考。

## 验证失败原因，不只是看它失败

这一点值得单独强调，因为它是最常被跳过的。

「我看到它失败了」和「我确认它因为正确的原因失败了」是两件不同的事。

**正确的失败**：测试失败，因为被测功能不存在（`AttributeError: module has no attribute 'new_feature'`、`AssertionError: expected X, got None`）——说明测试覆盖了正确的路径，继续写最小通过代码。

**错误的失败**：测试失败，因为测试本身有 bug（`ImportError`、fixture 名字打错、语法错误、断言写反了）——此时修的是**测试**，不是实现，修完之后**重新验证 RED**。

**危险的通过**：测试在 RED 阶段就通过了——意味着测试覆盖的是已有行为（不是新行为），或者测试根本没经过真正的被测路径。这种情况删掉，重新写。

不读失败输出、不确认失败原因，TDD 就退化成了「写完产码再写测试然后看一眼通过」——铁律的全部意义都在这个确认里。

## Constraint-Parity Trap：行为一致 ≠ 约束一致

> 回扣根2（契约即 SSOT）：test double 必须强制与真实后端**同样的约束**。

这是最隐蔽的测试陷阱，也是最常让「测试全绿」在生产出错的根因。

**核心问题**：内存 fake 能完美镜像真实后端的**行为**，但真实后端的**约束**它一个都没有——唯一索引冲突、外键级联删除、触发器强制的不可变性、并发写入时的行锁。内存 fake 对这些一无所知，所以也无法拒绝。

你的测试套件一片绿，但你从未在测试里触发过真实后端会拒绝的情况。这些不变式不是「已经测了，只是没有显式断言」——它们**没被测到**。

**处理原则**：

1. **约束类不变式对真实后端验**：关键的不变式约束（唯一性、外键完整性、并发安全），至少有一个测试用真实后端（数据库、消息队列的真实实例，在 CI 里跑 testcontainers 或等效的容器化后端）故意触发约束违反，看它正确地被拒绝。
2. **并发约束用真正的并发探针**：如果有并发安全要求，测试里用真正的并发（`asyncio.gather` / goroutine / thread pool）同时触发，而不是串行等待假装并发。顺序执行掩盖掉的竞态，并发探针一眼看穿。
3. **Mutation test 怀疑一个 test double**：对一个依赖 mock 的测试持怀疑态度时，临时故意在**生产代码**里引入 bug（翻转一个条件、注释掉一行），看测试是否仍然通过——通过的话，这个测试在测 double，不在测产码。

**行为一致 ≠ 约束一致**——这句话是根2在测试层的落点：test double 跟真实后端签的合约不只是「相同的返回值」，还包括「相同的约束执行」。

## Completion Gate：证据优于声称

> 回扣根5（证据优于声称）：「done」是关于世界的声称，要事后证据，不接自述。

「完成了」「全绿了」「已修复」——这些都是关于世界当前状态的**声称**，声称需要**证据**，而证据必须在**最后一次编辑之后**生成。

**Completion Gate 三步**：

1. **有范围的回归运行**：在最后一次改动之后，运行测试，完整读输出——不是截断版，不是 summary，是实际运行产生的日志里的 FAILED/ERROR 行。`grep -E "FAILED|ERROR"` 零命中 + Results 尾行摘要一致。
2. **质量 gate**：lint（代码风格检查）、静态类型检查（pyright / tsc / mypy 等，按项目配置）、格式检查——这些是**不同的检查**，缺一不可（格式通过 ≠ lint 通过 ≠ 类型检查通过）。
3. **报告观察到的事实**：日志说什么，报什么；跳过的步骤，**明说跳过了，原因是什么**，不用「其余部分应该没问题」代替。

**一个 worker 的自报「全绿」不是证据。** 有据可查的是：你在最后一次编辑之后实际运行了测试，读了输出，对输出有所引述。自述、感觉、「我相信应该通过了」——都不是证据。

整合者（orchestrator 或 reviewer）**不信任 self-report，独立重验**。这不是不信任 worker，而是方法论的一部分：verification 与 implementation 分离，才能给「green」这个词任何意义。

## 卡住时怎么读

不知道如何写测试，不等于跳过测试。

| 症状 | 读法 |
|---|---|
| 不知道怎么测它 | 先写出期望的 API 调用 + 断言——测试就是接口的设计草图，写测试倒逼出接口设计 |
| 测试的 setup 极其庞大 | 被测对象依赖太多——精简接口，或提取 fixtures；测试 setup 的重量是依赖复杂度的直接映射 |
| 必须 mock 掉一切才能测 | 被测代码耦合太紧——改成依赖注入（Protocol / interface），见 `engineering-with-craft` OOP 相关部分 |
| 测试极难写，感觉不可能 | 设计难以使用——这是一个值得讨论的设计 fork，不是「跳过测试」的理由 |
| 测试刚好就是很简单的东西 | 简单的代码也能崩。测试三十秒，生产 incident 不止三十秒 |

**硬写不出来的测试是设计在批评自己。** 听它，不要射杀信使。

两者不同 plane，互补不重叠。本文是手艺的纪律，`dev-as-ml-loop` 是循环的形状。

## Rationalization 表

以下每一条都曾被有能力的工程师（或 agent）在即将交付未验证的改动前真诚地说出。读这张表的最佳时机，是**你发现自己开始和纪律谈判的那个时刻**——谈判本身就是症状。

| 你对自己说 | 现实 |
|---|---|
| 「太简单了，不需要测试」 | 简单的代码也会崩。测试三十秒；production incident 可没有。 |
| 「我之后再补测试」 | 补写的测试会立刻通过，而你从未看着它失败过，所以它证明不了任何事。 |
| 「测试后写和测试先写，精神是一样的，不是形式」 | 测试后写回答「这代码做了什么」；测试先写回答「这代码应该做什么」。前者被实现偏置——你验证的是你记得的边界，不是你会发现的边界。 |
| 「我已经手动测过了」 | 临时验证 ≠ 系统化：没有记录覆盖了什么，代码下周改了之后不可重跑。 |
| 「删掉 X 小时的工作太浪费了」 | 沉没成本。时间无论如何都回不来；能选的是：一份从失败测试长出的、可信任的代码，还是一份反向补写的、在交付时点恰好通过的套路。未测试的可运行代码是穿着绿徽章的技术债。 |
| 「留着探索代码当参考再写测试」 | 你会适配它。那是测试写在后面，多走了几步。删掉，意味着删掉。 |
| 「我需要先探索一下」 | 完全合法——探索之后，把探索扔掉，从失败测试开始真正的实现。 |
| 「测试很难写，跳过」 | 难写的测试是设计在批评自己。听它（见上面「卡住时怎么读」）；不要射杀信使。 |
| 「TDD 会拖慢我」 | 比什么更慢——在生产里 debug？一旦把 incident 时间计入，红绿重构才是快路径。 |
| 「这块现有代码本来就没有测试」 | 你是在改善邻里，不是在匹配它。你动到的部分，加测试。 |
| 「这次不一样，因为……」 | 不一样。这个句式本身就是红旗。 |
| 「我用 mock 测过了，跟真实后端行为一致」 | 行为一致 ≠ 约束一致（见 Constraint-Parity Trap）。内存 fake 没有 unique index，没有 FK cascade，没有触发器——你从未触发过那些保险丝。 |

## 停下重开的红旗

以下任何一条在你自己行为里出现，意味着循环已经断掉：

- 产品代码存在，对应的测试不存在。
- 一个新测试第一次运行就通过了（而你没有删掉它重写）。
- 你无法解释 RED 运行**为什么**失败（或者你根本没读日志）。
- 「测试在后续 commit / PR 里补」。
- 你在修改测试来让它匹配代码——而代码才是那个在怀疑中的东西。
- 你准备报告「全绿」，但最后一次编辑之后你没有运行过任何测试。

**恢复路径永远一样**：删掉未测试的产品代码，写失败测试，诚实地重新开始循环。
