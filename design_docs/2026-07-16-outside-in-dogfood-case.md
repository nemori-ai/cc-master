## 主案例：watchdog 唤醒机制的承重假设，被用户外部知识推翻

### 原假设（agent 推断，无外部接地）

编排开局武装 watchdog 时（board log `2026-07-16T16:45:05Z`）：

> 「watchdog 已 arm：recurring cron `bf071ae1`（每~28min recon）+ 一次性 deadline 锚点 `dfc84cbf`（T-3h）/`5d3bf979`（T-90min）；wave1 已派发……」

承重判断藏在这一步里：**「cron 定时唤醒足够可靠，可以充当 watchdog 的主机制」**。它的支撑**只有内部推断**——「工具存在（`CronCreate` / cron）→ 默认它可靠」，既没查运行时事实、没读权威文档、也没问用户。按证据五分级，这是一条典型的 **agent 推断**（危险级）冒充「已知事实」。

它**承重**：watchdog 的全部价值就是补「静默失败盲区」——后台任务 hang / 静默死 / phantom 时把 orchestrator 唤回来 recon。如果唤醒机制本身不可靠，整张安全网是空的，而且**它的失效方式恰恰是静默的**（到点没被唤醒，你不会收到任何报错）。门控大：一整场 12h 冲刺的活性监控都押在它上。

### 证据分级 + 推翻它的证据来源

- 原假设级别：**agent 推断**（内部唯一支撑）。
- 推翻它的：**外部证据**——用户的一句运营知识 tip。board log `2026-07-16T16:49:25Z`：

> 「watchdog 双轨升级（**用户 tip：cron 唤醒不保证，background shell 是可靠 floor**）：主轨=background shell `bb0ln7p7o`（sleep 至 18:40Z=wave1 p95）；附加轨=cron `bf071ae1`/`dfc84cbf`/`5d3bf979`。此后每次唤醒时 re-arm 下一段 shell floor」

「哪一项外部事实最能推翻它」的答案，就是「cron 在这个环境到底可不可靠」这条运行时 / 运营事实——它**不可能**由内部推断或再派一个 subagent 复读得到，只能来自外部（实测 / 权威文档 / 懂这台机器的人）。

### 校准手段与成本

- 落在校准手段成本阶梯的**第 2 档（用户澄清）**：一次前台 tip，成本近乎零（用户主动给的）。
- 这条假设属于运行时事实，理论上也可走**第 1 档（实测：真跑一次 cron 看会不会准点触发）**或**第 6 档（权威资料：查该唤醒机制的送达保证）**——但用户 tip 是当时最快落定它的外部手段。**手段配假设类型**在这里体现得很干净：「cron 可不可靠」这类环境事实，问用户 / 实测 / 查文档能答，纸上推演答不了。

### 修正动作

不是「删掉 cron」，而是**双轨降级**：主轨换成 background-shell floor（可靠、`until` 轮询），cron 降为附加轨；此后每次唤醒 re-arm 下一段 shell floor。这正是 orchestrator 手册里「唤醒工具按情境降级，**background-shell 轮询永为 universal floor**」那条降级链纪律的实战落地——外部证据把一个「工具存在即可靠」的乐观假设，纠正回「floor 必须是保证可靠的那一档」。

### 这个案例为什么正中 outside-in 命门（诚实标注）

关键的诚实点：**这次是用户主动 tip、不是 orchestrator 自己跑了 outside-in 检查抓到的**。假设当时确实是「承重 ∧ 内部推断唯一支撑 ∧ 门控大」三条全中——正是纪律该触发校准的情形——但 orchestrator 没有自发问「cron 到底可不可靠」，是外部知识恰好撞了进来。

这**恰恰是 issue #142 要植入纪律的理由**：如果用户没 tip，这场编排会带着一张**静默失效的安全网**跑完整个冲刺，而 watchdog 的失效本身是静默的——你不会在验收时发现它，因为它要兜的就是「你没发现的失败」。端点验收兜成品对不对，兜不回一个「监控机制本身建在未验证假设上」的方向错误。把「我哪个承重假设只由内部推断支撑」变成决策程序里一等的、随 reinject 留在 context 的检查项，就是要让这种校准**不必靠运气**。

## 佐证案例（同一纪律、orchestrator 自发的一侧）：kimi 配额「unsupported」文档假设不被采信

同一场编排里有一个反向的正例——orchestrator **自发**拒绝了一个文档层假设。用户离线前追加需求时，orchestrator 的记录（board log）：

> 「用户追加需求（离线前最后消息）：kimi 配额查询须单独调研（大概率有解，最差 statusline 定制）→ K9 已立，K5 增依赖 K9」

这里被拒绝的是「kimi 配额查询大概率 unsupported」这一**文档 / 直觉层假设**。处置不是「接受 unsupported、按没有配额信号规划」，而是**立一个真实校准节点（K9）去实测**——把一条「待验证假设」操作化成一个 board DAG 节点（正是证据五分级里「待验证假设 = pending 判断 ＋ 一个真实校准节点」的用法）。K9 的研究**交付有效**（board 的安全事件 jc 附带确认「研究交付有效」）。

对比价值：主案例是「承重假设靠外部证据被动纠正」（说明盲区代价），佐证案例是「文档层 unsupported 假设被主动送去实测校准」（说明纪律自发跑起来长什么样）——一被动一主动，同一根走廊，都不是闭门造车。

> 证据边界（忠实性）：本文只引用 board `log` / `jc` 里可复核的原文与时间戳；K9 实测到的具体配额接口路径不在本 board 的 log 摘要中，故本文只述「文档层 unsupported 假设被送去实测且交付有效」这一层，不复述未在 board 落痕的接口细节。
