VERDICT: NEEDS-ATTENTION

FINDINGS:

1. **[critical] §7.1, §10, §12.2 — No cross-version capability gate.** The plugin and `ccm` release independently, but the design reuses `goal-contract/v1` without adding a deadline capability. A new plugin paired with an older `ccm` would pass today’s bootstrap precheck, fail or ignore `goal deadline set`, and receive `ok` from the old goal checker, permitting dispatch with no DDL confirmation. Suggested fix: introduce `delivery-deadline/v1` plus a versioned goal-check contract, require it before arming on all three hosts, and specify both old-plugin/new-ccm and new-plugin/old-ccm behavior.

2. **[major] §2.4, §3.2, §7.1 — `asserted` unlawfully passes the dispatch gate.** The design maps `asserted` to a usable deadline and lets `goal check` return `ok`, while issue #149 requires explicit deadline/no-deadline confirmation before goal check or DAG dispatch. This also contradicts §7.1’s own statement that confirmation precedes `ok`. Suggested fix: only `confirmed` and `none` settle the deadline gate; agent-inferred dates remain pending. An explicit `--ddl` may become confirmed immediately only if the contract defines the flag itself as the user’s confirmation.

3. **[major] §2.1, §3.1, §3.3, §5.2, §12.2 — Deadline revision and transition auditing are missing.** The issue requires every add, confirmation, and change to retain revision, reason, and timestamp, but the proposal deliberately avoids both goal and deadline revisions, while §5.2 nevertheless fingerprints a nonexistent “DDL revision.” Moreover, the current `goalAmend` mutation reconstructs `goal_contract`, so it will drop the nested deadline unless explicitly changed. Suggested fix: add a monotonic deadline revision—or make every deadline mutation a Goal Contract revision—plus reason, actor, timestamp and goal-revision binding; define the complete transition table, including deadline↔no-deadline changes, and update `goalAmend` to preserve and revalidate the field.

4. **[major] §3.4 — Date-only input is converted into a potentially wrong UTC deadline.** Turning `2026-07-20` into `2026-07-20T23:59:59Z` invents both timezone and delivery semantics; for a Beijing user intending local end-of-day it shifts the deadline by eight hours. Deferring the convention to D2 does not satisfy the issue’s requirement for unambiguous absolute time, timezone, and precision. Suggested fix: keep date-only evidence pending until time and timezone are confirmed, or represent an explicit timezone-aware interval instead of silently choosing UTC end-of-day.

5. **[major] §2.4, §7, §9 — The new `deadline_pending` verdict lacks a complete consumer migration.** Current reinjection code validates a closed verdict set and would translate the new verdict into `check_unavailable`; verify-board has a different closed set and can report it as an integrity failure. The design mostly names bootstrap and identity-nudge, omitting reinject, verify-board, command/skill prose, schemas, tests, and their three-host contracts. Suggested fix: inventory every `goal check` consumer, version its output schema, update `using-ccm` in lockstep, and require N-host contract tests for each affected hook—not only identity-nudge.

6. **[major] §5.3–§5.5 — “Direct injection plus durable exactly-once visibility” is not implementable with the current inbox protocol.** A risk hook can directly inject a warning and append a fresh notification, after which the separate coordination-inbox hook sees that unseen ID and injects it again; the proposed fingerprint does not coordinate delivery between independent hook processes. Suggested fix: choose one delivery path, or persist a session-epoch delivery receipt that the inbox understands and suppresses; support deterministic notification IDs and update the coordination-inbox CONTRACT for all hosts.

7. **[major] §4.3, §6.1 — The throughput fallback is mislabeled as resource-aware and conservative.** The existing throughput Monte Carlo samples historical daily completion and scales by `effective-n`; it does not schedule the current DAG against board WIP, owner WIP, worker skills, harness capacity, or resource contention. It can therefore produce a false-green probability precisely where the requirement forbids one. Suggested fix: do not emit a resource-aware probability unless RCPSP or an explicitly modeled capacity scheduler participates; otherwise return `unknown` or a separately named throughput heuristic that cannot yield green.

8. **[major] §6.2, §9, §12.1 — Calibration/backtesting is the most underestimated 12-hour item.** Existing boards generally lack historical deadlines, observed final-delivery outcomes, and event-sourced snapshots of dependencies, estimates, WIP and task states. The current `--as-of` behavior cannot reconstruct those facts, so meaningful Brier scores and reliability diagrams cannot be obtained merely by replaying current boards. Suggested fix: make a data-availability spike the first acceptance gate, distinguish synthetic scheduler validation from empirical calibration, and ship v1 as explicitly uncalibrated—or defer probability-based hook bands—until labeled snapshots exist.

9. **[major] §4.2, §4.4 — “Global acceptance incomplete” has no canonical predicate.** “Board not archived” and “has unfinished tasks” are not equivalent to incomplete delivery: a board can remain active after all tasks finish, or be archived for handoff while delivery remains incomplete. This can suppress genuine overdue warnings or repeatedly warn after acceptance. Suggested fix: define an engine-owned delivery-completion predicate or explicit acceptance marker, with documented interactions for archive, handoff, resume, and legacy boards.

10. **[major] §7.1, §10 — `--goal` evidence preservation is asserted but not designed.** Current bootstrap interfaces chiefly treat the goal as free-form positional text; the existing command contract does not define `--goal` parsing. Adding `--ddl` without a shared grammar risks consuming flag text as goal evidence, losing provenance, or diverging among Claude Code, Codex, and Cursor. Suggested fix: specify the complete bootstrap argument grammar, precedence and conflict rules, evidence provenance, escaping behavior, and identical contract tests for positional and `--goal` forms on all three hosts.

11. **[major] §2.1, §2.3 — Reserving `kind: soft` creates a valid but undefined state.** The schema would accept `soft` while v1 consumers intentionally ignore the distinction, allowing imported or manually edited boards to carry a soft deadline that risk and overdue logic silently treat as hard. Suggested fix: constrain v1 to `kind: hard` or omit the field; add `soft` only with defined behavior and a schema/capability version bump.

12. **[minor] §3.5, §5.3 — Repository details have already drifted.** The design says lint rules grow from 83 to 86, while the current lockstep documentation reports 82, and the proposed advisory message uses imperative wording such as “immediately prioritize,” conflicting with ADR-018’s advisory/directive distinction. Suggested fix: derive rule counts mechanically and either soften the wording to a recommendation or classify and authorize it as a directive.
