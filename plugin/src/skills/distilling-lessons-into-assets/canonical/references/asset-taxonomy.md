# 四类资产的定义与判据

四类资产不是同一种东西的四个模板,是四种**不同性质的知识**各自的天然容器。选对容器,经验才活得下去。

<!-- ccm:k:start point:distill.taxonomy -->
## 纪律文档(discipline document)

项目里那份大家默认会去读的主纪律载体——常见形态是一份贡献者入门文档、一份设计文档、一份变更记录。

**适合承载**:
- 一次性但**意义持久**的事实(这次因为某个具体原因踩了坑,下次要注意这件具体的事)。
- 项目专属的红线或惯例(这个项目就是这么定的,换个项目不一定适用)。
- 需要一句 **why** 才站得住的具体约束(单纯写"不要做 X"没有说服力,写"不要做 X,因为上次 X 导致了 Y"
  才会被人记住)。
- 归宿判不清时的**托底选项**——落地成本最低,错了也最容易改。

**不适合承载**:
- 需要被反复不同任务触发调用的可迁移判断力——纪律文档是线性阅读的,一段判断力埋在文档中部,下次压力
  大的时候没人会想起来翻到那一段去看。判断力需要靠"触发描述"主动被找到,而不是靠"记得曾经读过"。

## skill(带触发描述、可独立调用的判断力文档)

一份带触发描述、可以被独立识别并加载的方法论/判断力文档。

**适合承载**:
- 可复用到**其他任务**的判断力或方法论——不是"这次踩了这个坑",而是"一套遇到同类情况时该怎么判断的
  方法"。
- 需要专门触发描述才能被准确找到的内容——如果这段内容只会在读者线性通读时被看到,它就不该是 skill。
- 压力下容易被合理化掉的规则——如果这条规则的价值恰恰在于"agent 在赶时间/有借口的时候还是要守住它",
  它需要纪律型 body(带 Rationalization Table),而纪律型 skill 有自己的压力测试要求,不是写完就完事。

**不适合承载**:
- 一次性事实(该进纪律文档,不需要"触发"机制,读一遍记住就够了)。
- 纯确定性的机制形状(该进 workflow——机制不需要判断力,不该占用"skill = 判断力容器"这个位子)。

## workflow(确定性编排脚本)

一段把"先做什么、并行还是串行、怎么 fan-out、怎么收敛"这类**结构**固化下来的脚本或流程定义。

**适合承载**:
- 确定性的编排/并行形状本身——这一步和那一步谁先谁后、要不要拆成多个并行分支,这些是结构问题,一旦定
  下来不需要每次重新判断。

**不适合承载**:
- 需要临场判断力的决策点——如果某一步"具体怎么做"要看情况判断,那个判断力应该留给触发这段
  workflow 的 agent 去做,不要把判断逻辑硬编码进脚本里假装是确定性的。把判断力塞进 workflow,会让它在
  遇到脚本没预见的情况时要么僵化出错、要么被迫在脚本里堆砌越来越多的 if-else 来模拟判断力——不如老实
  把判断力留在 agent 侧。

## subagent(角色化的专职人格定义)

一个有独立 persona、独立工具/上下文边界、会被反复以同一角色调用的命名分工。

**适合承载**:
- 需要角色化专职视角的分工——这个角色总是从同一个视角看问题、总是只被授予这一类工具、总是被反复以这个
  身份召唤。

**不适合承载**:
- 一次性任务委托——普通派发就够了,不必为了一次性的事专门新建一个命名 subagent 定义,那只会制造出一堆
  只用过一次、无人维护的角色定义文件。

## 一句话判据

**事实 → 纪律文档;可复用判断力 → skill;确定性结构 → workflow;专职角色 → subagent。** 拿不准先回到
[routing-decision-tree.md](routing-decision-tree.md) 的顶层三问重新过一遍,不要凭"这条经验感觉多重要"
去猜归宿——重要程度和归宿类型是两个独立的维度。
