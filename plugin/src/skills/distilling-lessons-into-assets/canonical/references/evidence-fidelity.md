# 证据忠实性:泛化 vs 收窄

本 skill 唯一的硬约束:**候选内容草稿到写进目标资产之间,不允许发生一次会脱离原始证据的改写/泛化。**
这条细则讲清楚"泛化"具体长什么样、遇到措辞不够用时该怎么**收窄**而不是**泛化**,以及"绝不静默丢弃"
在证据强度这个维度上具体怎么落地。

## 什么是"脱离证据的泛化"

泛化是把一个**有场景限定**的观察,写成一个**没有限定**的普遍规则。典型模式:

> 证据:这次因为某个具体原因踩了坑,下次注意这件事。
> 泛化后的错误写法:"总是应该避免这类做法。"

"这次因为 X 原因踩了坑" 是一个受限于具体场景(这一次、因为这个原因)的观察;"总是应该避免 X" 抹掉了
场景限定,变成了证据不支持的普遍论断——证据只证明了"这一次、这个原因下会出问题",没有证明"任何时候
做这件事都会出问题"。

再举一例:一条候选经验只有一次实例支持("这一次这样处理解决了问题"),落地时写成"这套方法总是有效"
同样是泛化——单一实例的证据强度撑不起"总是"这个量词。

## 怎么收窄而不是泛化

措辞觉得不够有力、想让它听起来更普遍适用时,**先问:证据到底支持多强的论断?** 然后把措辞收窄回证据
能站得住的范围,具体做法:

- **保留场景限定词**——"在某场景下观察到某结果"优于"应该总是怎样"。场景限定词不是软弱,是诚实。
- **保留实例计数**——证据是"三次独立实例都指向同一个修复方式" vs "只观察到一次",落地措辞应该分别
  体现为"这套修复方式在多次独立场景下都成立,值得作为通用做法"与"这一次这样处理解决了问题,尚不确定
  是否普遍适用"——实例计数直接决定了措辞的确信程度。
- **不要为了"听起来更有分量"而删掉限定词**——删掉限定词不会让证据变强,只会让论断变得证据不支持。

## 合并多条候选经验时的边界

合并同落点的多条候选经验、让措辞连贯衔接是允许的——比如把三条独立实例的候选经验合并成一句"该修复方式
在三次独立场景下都得到验证",这不是泛化,这是如实反映了合并后的证据强度(三次独立实例本身就是比单次
更强的证据)。**不允许的是**合并过程中引入任何一条候选经验本身都不支持的新论断——合并是让已有证据说
话说得更连贯,不是借合并的名义夹带私货。

## 发现候选草稿本身已经泛化过头

来源文档里生成的候选内容草稿,有时候在生成阶段就已经犯了泛化的错误(把一次性观察写成了普遍规则)。
遇到这种情况,**不是照抄就完事**——按证据范围把措辞收窄回去,并在落地时显式注明:

> 已收窄:原候选草稿过度泛化,落地时按证据范围收窄为「…」。

这条注明本身也是给人工审阅一个可核对的锚点——审阅者应该能看到"这里发生过一次收窄,原因是什么"。

## 证据强度与"绝不静默丢弃"的关系

弱证据(模糊、缺少可复核的具体来源、泛泛而谈的观察)**不是丢弃这条候选经验的理由**——弱证据是"用更
保守的措辞、更低调的归宿(纪律文档式的观察记录而非升级成 skill/workflow)、并显式标注置信度低"的理由。
真正应该发生的是:**弱证据 → 收窄措辞 + 标注低置信 + 落地成本最低的归宿;不是 弱证据 → 直接不处理**。
让人工审阅环节看到"这条证据弱,已收窄且标注",审阅者可以选择否决它,但否决权在人工审阅,不在蒸馏阶段
的规划环节。
