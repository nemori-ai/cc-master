# OBJECTIVE — distilling-lessons-into-assets

J_top: 蒸馏流程的执行 agent 拿到一批候选经验时，用本 skill 的三问决策树把每条经验路由到正确的资产类型
（纪律文档 / skill / workflow / subagent），落地时套用该资产既有文风、合并同落点候选、且写入内容不脱离
原始证据泛化；遇到二义性显式标双落点，遇到归宿不确定或目标基础设施缺失时落最低成本的纪律文档托底并
显式标注，绝不静默丢弃任何一条候选经验（哪怕证据弱）。

baseline_reference:
  user_task: 给 agent 一批候选经验（覆盖一次性事实、单实例观察、模糊无证据观察、多实例可迁移判断力各
    至少一条），在时间压力下要求路由到四类资产并起草落地措辞。
  without_skill_floor: 已跑一轮真实 baseline（记录见 .design/DESIGN.md §2）——大多数路由判断默认表现
    尚可（不会把重要一次性事实误判成 skill、不会把单实例过早固化成 workflow），**但对"证据弱/模糊/无
    具体来源"的候选经验，默认行为是直接丢弃、不留痕迹**——这正是本 skill 要堵的缺口：弱证据不是不处理
    的理由。
  expected_uplift: 有本 skill 后，弱证据候选经验被收窄措辞 + 标注低置信 + 落最低成本归宿，而不是消失；
    合并多实例候选时的措辞强度与证据强度对应更一致；归宿不确定/无基础设施时显式标注而非隐性吞掉。

strict_dims: [绝不静默丢弃候选经验（弱证据/归宿不确定/无基础设施都走"标注+托底"而非丢弃）, 写入内容
  的证据忠实性（不脱离证据泛化，措辞强度与证据强度匹配）]

rationale: 本 skill 是 Craft C 纪律型——含压力下会被合理化掉的规则（"这条经验挺重要该进 skill”“证据
不够就写通用点”“没地方放就别提了”“证据太弱干脆不处理”），故走 pressure baseline 而非仅框架连贯性
验证。两条 strict_dims 对应 baseline 实测暴露的真实缺口（丢弃）与 spec 明文的唯一硬约束（证据忠实性），
符合本仓"1-2 个 strict_dims"约束。

## 非目标（notes）

不要求 agent 记住四类资产的每一条判据原文——决策树的三问框架才是货，taxonomy 表是速查脚手架。不评判
"资产落地机制本身操作对不对"（命令怎么敲、字段怎么填）——那不是本 OBJECTIVE 的成功条件，归该载体自己
的操作手册。不与"这个能力该不该独立成 skill"的治理判断相干——那是不同读者、不同自包含要求的治理层
判断，本 skill 的 J 不覆盖它。
