# distilling-lessons-into-assets — 设计宪法（DESIGN.md）

> 本仓 DESIGN.md 统一 6 段（`curating-skill-portfolios` 约定）。SKILL.md 实质改动先在此更新。

## 1. One-liner

给"把候选经验蒸馏成项目资产"这一步提供品味层判断力：一条候选经验该落成纪律文档 / skill / workflow /
subagent 中的哪一种、以及落地到该资产时该怎么写才不走样。命令体只管流程骨架（认参数、读文件、fan-out、
PR），本 skill 承载判断力内容——呼应本仓既有"命令薄、skill 承重"分层。

## 2. Craft 自分类

**Craft C 纪律型**（process-control 强，含压力下会被合理化掉的规则）：body 含 Rationalization Table +
唯一硬约束（证据忠实性）+ 决策树的兜底纪律（不确定/无基础设施绝不静默丢弃）。走 pressure baseline 闸。

- **Pressure baseline 记录**：起一个不带本 skill 的 general-purpose 子 agent，在时间压力下给它 6 条
  候选经验（含一次性事实、单实例观察、模糊无证据观察、多实例可泛化判断力各一批），要求路由到四类资产
  并起草落地措辞，再自我复盘。观察结果：该次 baseline 整体表现优于预期——未过早升级单实例观察为
  skill/workflow、未把重要的一次性事实误判为 skill、正确识别了唯一满足"多实例 + 可迁移判断力"门槛的
  条目该进 skill。**但暴露了一个真实缺口**：面对一条"模糊、无具体证据、感觉像放之四海而皆准的真理"的
  候选经验，baseline agent 选择直接丢弃、不留任何痕迹——这正是本仓"绝不静默丢弃"纪律要堵的那类失败：
  弱证据不是不处理的理由，是"标注低置信 + 收窄措辞 + 落最低成本归宿"的理由。该发现已写入
  `references/evidence-fidelity.md`（末节）与 `references/landing-craft-by-asset-type.md`（Rationalization
  Table 新增两行：弱证据丢弃 / 单实例过早固化）。次要观察：baseline 对"合并多实例候选经验时该不该收紧
  措辞"处理正确（用"多次独立场景验证"而非夸大成"总是有效"），说明证据强度→措辞强度这条对应关系本身不
  难，真正的缺口是"弱到令人想放弃时该怎么办"，不是"措辞尺度怎么把握"。

## 3. Value triad（三视角价值）

### 3.1 Plugin 视角

补上"经验→资产"这一步此前完全缺失的品味层——没有它，任何蒸馏流程要么靠调用者临场发明归宿判断（不
一致、不可复用），要么把判断力硬塞进流程命令体本身（命令体膨胀、丧失"命令薄、skill 承重"的分层）。

### 3.2 Agent 视角

给蒸馏流程的执行 agent 一套可复用的路由判断：先问"事实还是判断力"，再问"机制还是判断力"，再问"要不
要专职角色"；遇到二义性允许双落点，遇到不确定/无基础设施用最低成本归宿兜底、绝不让经验消失；写入时守
住证据忠实性这条唯一硬约束。

### 3.3 Human 视角

蒸馏出的项目资产更少重复、更少走样、更少因为"没地方放"而丢失——审阅者拿到的是已经过品味筛选、标注了
置信度与收窄痕迹的候选，审阅负担更低。

## 4. 责任边界

### 4.1 IN scope

候选经验→资产的路由判断（四类资产判据 + 三问决策树 + 二义性/不确定兜底）+ 每类资产的落地手艺与反模式
+ 证据忠实性硬约束（泛化 vs 收窄、绝不静默丢弃）。

### 4.2 OUT of scope（明确移交给谁）

| 不归本 skill | 归谁 |
|---|---|
| 代码本身（领域/类/合约/测试）该长什么样 | 另一个专管工程手艺的判断力（不同 plane：本 skill 管"经验→资产"路由，不管代码设计本身） |
| 把目标切成任务依赖图 | 另一个专管目标分解方法论的判断力（输入输出形态完全不同：目标 vs 候选经验清单） |
| 把已切好的任务优化到验收 | 另一个专管执行侧循环心智的判断力（不涉及任务执行循环） |
| 资产落地机制本身怎么操作（命令行怎么敲、字段怎么填） | 该载体自己的操作手册 |
| "这个能力该不该独立成 skill"的治理判断 | 仅服务本项目维护者的治理层判断，读者与自包含要求都不同，本 skill 不引用它 |

### 4.3 Boundary heuristic（一句话判定法）

"一条候选经验该归属哪类资产、该怎么落地才不走样" → 本 skill；"代码/目标/任务本身该怎么设计/切分/推进"
→ 各自专门的判断力；"这个能力该不该独立成 skill" → 项目自己的治理规范（若存在）。

## 5. 触发与反例

### 5.1 Recognition cues

拿到一批候选经验要决定归宿；判断一条经验该进纪律文档还是 skill；判断该不该固化成 workflow 还是留判断
力给 agent；判断该不该专门配一个角色；候选内容草稿的措辞是否脱离了原始证据；证据弱/归宿不确定时怎么
处理才不让经验消失。

### 5.2 Counter-examples

代码本身该怎么设计（领域/类/合约/测试）；把目标切成任务依赖图；把已切好的任务推进到验收；ccm 类命令
怎么敲、字段怎么填；"这个能力该不该独立成 skill"的项目治理判断（不同读者、不同自包含要求）。

### 5.3 Pre-flight gate（纪律型专属）

- 改 Rationalization Table / 决策树兜底纪律 / 证据忠实性细则前，先跑一轮 pressure baseline 看会不会
  重新暴露已知缺口或新缺口。
- `description` 改动前后跑 Track A trigger eval，尤其确认"代码工程手艺"“目标切分”“任务执行”类
  query 不误触发。

## 6. 演化锚

- **蒸馏流程本身发现新的归宿判断盲区**（如遇到一类经验四问都答不出、或遇到新的资产类型）→ 先补决策树
  兜底，再考虑要不要扩展四类资产taxonomy（新增第五类资产前先证明前四类真的容不下）。
- **新的自我说服模式被观察到** → 补进 Rationalization Table，同类改动走 pre-flight gate。
- **证据忠实性判断出现新的边界案例**（如多来源冲突证据怎么合并）→ 补进
  `references/evidence-fidelity.md`，不要另开新文件制造分裂。
