1. **后台 terminal session（当前 agent runtime 已验证）** —— 适合同一 session 内的长跑 shell / watch loop：启动后拿真实 session id，后续 poll 输出与退出码；再用 `--mechanism shell --job-id <session-id>` arm，并把命令、工作目录、predicate、取消方式写入 checklist。它是可用的 agent 能力，不是“用户产品面按钮”。
2. **Codex app thread automation（若当前环境可用）** —— 官方支持 attached-to-thread 的 recurring wake-up，适合检查长命令、轮询连接源、继续 review loop。当前 agent runtime / CLI 没有稳定 automation 创建命令；若用户环境提供并已创建 Codex app automation，拿真实名称 / id 后用 `--mechanism loop --job-id <automation-id>` arm。
3. **外部 scheduler + `codex exec resume`（脚本化 floor）** —— cron、systemd timer、CI scheduler、GitHub Actions 等调用 `codex exec resume <SESSION_ID>` 或 `codex exec resume --last`，提示它 recon board、检查 predicate、更新 task。创建成功并拿 scheduler id / run URL 后，用 `--mechanism cron --job-id <handle>` arm。适合跨 session 无人值守，但这是外部调度，不是 Codex-native in-thread timer。
4. **Codex cloud status loop** —— 对 offloaded cloud work，记录 `codex cloud exec` 返回的 task id，后续用 `codex cloud status/list/diff/apply` recon；用 `--mechanism monitor --job-id <cloud-task-id>` arm，调度这个 recon 靠后台 terminal、thread automation 或外部 scheduler。
5. **manual recon（诚实兜底）** —— 没有真实 wakeup handle 时，**不要 arm watchdog**；把下一次检查时间、predicate、负责人、取消条件记进 blocked / recon 状态与 log，等待可续事件。不要伪造自动唤醒。
